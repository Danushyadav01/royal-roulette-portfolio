The Problem

Casino interfaces are broken
for the modern web.

Casino and betting interfaces are notoriously cluttered, overwhelming for beginners, and rarely adapt smoothly across mobile and desktop in real time.

The challenge: design a platform that feels luxurious, readable, and functional — simultaneously for the host running the table and the player placing bets.
📱
Responsive Chaos
Roulette boards collapse on vertical mobile screens. Chip click targets become impossible to hit accurately.
🧠
Cognitive Overload
Existing casino UIs bombard beginners with too much data at once — balance, round history, live chat, and betting controls fighting for attention.
⚡
Latency Disconnect
Real-time sync between host wheel spin and player bet locks needed near-instant UI feedback to feel trustworthy.
◆
User Architecture

Two users. Two worlds.
One platform.

User Type A
The Host
Table Administrator
  • Needs full game control — create table, set limits, start spins
  • Requires live view of all connected players and their bets
  • Needs automatic payout calculations after each round
  • Must be able to manage chips and track chip balances per player
  • Demands a clean admin dashboard — no clutter, high clarity
  • Needs session codes to share with players for joining
User Type B
The Player
Active Bettor
  • Joins via a session code — no account or login required
  • Needs distraction-free betting: chip selection → board placement
  • Must clearly see their balance and current round bets at all times
  • Requires visual feedback: wheel animation, win/loss result states
  • On mobile: one-thumb operation must be fully viable
  • Needs clear bet lock indicator when host has initiated spin

Design Decision · Mobile Layout
The Vertical Screen Problem

On a vertical mobile screen, a traditional roulette board is too wide and the numbers too small to tap accurately. The solution: stack the wheel above a scrollable number grid, and anchor the chip selector as a fixed bottom strip — eliminating accidental mis-taps through spatial separation.

Desktop Horizontal Mode

On desktop, the wheel moves left alongside the board in a split-panel layout. The chip bar becomes a sidebar column. Both modes share the same component tree — only the CSS grid breakpoints shift, ensuring zero logic duplication between builds.

Near-black with warm undertone. Mimics the velvet of a real casino floor. Reduces eye strain during extended sessions.

Warm gold over cool yellow. All action-critical elements (CTA buttons, selected states, win indicators) use this — trained recognition in 2 seconds.

Deep green for the betting board. Universally recognizable as the roulette surface — instant context without onboarding.
Rouge #A02020
Muted red — not alarm-red. Used for red number cells and the highest-value chip. Communicates risk without panic.

Key Learnings

What 3 months of
deep iteration taught.

01
Design the edge states first
Locked bets, disconnected peers, zero-balance states — designing the error and empty states before the happy path saved hours of rework. The UI is stress-tested before it's styled.
02
P2P sync changes your IA
When there's no server to arbitrate state, the host becomes the source of truth. Every UI decision — who controls what, in what order — had to reflect this P2P topology at the information architecture level.
03
Luxury ≠ complexity
The cinematic casino aesthetic came from restraint — fewer colors, slower animations, generous spacing. Removing elements (not adding them) was consistently the move that elevated the feel from "game" to "experience."
04
Build for the worst device
Testing on a low-end Android with a 5-inch screen exposed chip tap-target failures early. Designing for the worst case meant the premium experience was already solved on every device above that baseline.
05
Code literacy shapes design
Knowing CSS animation timing functions let me design the wheel deceleration curve in Figma with realistic physics — because I understood what was buildable before I drew it.
06
Feedback loops must be instant
A 200ms delay on chip placement felt like a glitch. A 100ms felt snappy. Designing micro-response timing — not just visual states — is a craft in itself that high-fidelity prototypes can't capture; it needs production code.
